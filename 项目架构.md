# Wuwa 3C 纵向切片项目架构

> 本文档是 `3c复刻.md` 的独立架构补充，不修改原文档。它以 Day 1～Day 4 和 Day 5 当前检查点为基线，规划后续所有功能如何设计与组织。

## 1. 架构目标与边界

### 1.1 项目目标

四周内完成一个可打包、可演示、可继续扩展的第三人称动作游戏 3C 纵向切片，覆盖地面移动、空中移动、探索动作、锁定与镜头、战斗、弹刀和双角色切换。

架构优先保证：

- **解耦合**：输入、动作规则、物理移动、战斗判定、动画表现、镜头和 UI 等各自有明确职责。
- **数据驱动**：角色差异、动作规则、数值、资源引用和镜头参数由 Data Asset、Curve 或 Data Table 配置。
- **单一权威**：同一状态或位移只允许一个系统拥有最终决定权。
- **可恢复**：每个临时动作都有统一进入、退出、取消和强制清理路径。
- **可观测**：动作拒绝原因、活动标签、移动模式、窗口与目标评分均可调试。
- **适度设计**：四周内保持单 Runtime Module，不引入完整 GAS、复杂 DI 容器或通用状态机框架。

### 1.3 明确不做

- 不按功能拆成多个 UE Module；四周项目中编译与依赖维护收益不足。
- 不引入完整 GAS；使用 Gameplay Tags、组件和轻量动作路由即可。
- 不让动画蓝图、Montage 或 Anim Notify 直接成为 Gameplay 状态权威。
- 不建立可脚本化的“万能动作系统”；只抽象项目中真实存在的共同生命周期。
- 不让 `AWuwaCharacter`、`UWuwaActionRouterComponent` 或 `UWuwaCharacterDefinition` 演变为包含所有逻辑的上帝对象。
- 不实现手柄、触摸或移动端输入。当前交付平台只支持 Windows 键盘与鼠标；模板遗留的设备配置不代表项目承诺兼容。

## 2. 总体分层与依赖方向

```text
键盘鼠标输入 / AI 决策
        │
        ▼
Input Adapter（PlayerController）
        │  生成语义化 Input Command
        ▼
Action Router + Input Buffer（规则与仲裁）
        │  产生获准的 Action Request / Action Context
        ├──────────────┬───────────────┬───────────────┐
        ▼              ▼               ▼               ▼
Locomotion         Traversal         Combat         Party/Targeting
物理与速度         环境查询与探索     命中与伤害       角色与目标上下文
        │              │               │               │
        └──────────────┴───────┬───────┴───────────────┘
                               ▼
                    Gameplay Read Model / Events
                         │                 │
                         ▼                 ▼
                    Animation          Camera / UI / VFX
                    只读表现             只读表现与反馈
```

依赖约束：

1. 表现层可以读取 Gameplay 状态或订阅事件，不能反向直接修改核心状态。
2. 功能组件之间优先通过窄接口、请求结构和事件通信，禁止相互保存大量具体组件引用。
3. Action Router 只做“能否执行、缓存、切换、结束原因”，不实现跑墙物理、攻击 Trace 或镜头插值。
4. Movement Component 是 Capsule 物理位移权威；Root Motion/Motion Warping 动作必须经过明确的位移策略选择。
5. Gameplay Tags 用于描述和查询事实，不替代事实拥有者。状态变化必须由拥有者 API 完成。

## 3. 核心运行时对象及职责

### 3.1 框架对象

| 对象                       | 负责                                                    | 不负责                          |
| -------------------------- | ------------------------------------------------------- | ------------------------------- |
| `AWuwaGameMode`            | 默认 Pawn/Controller、演示关卡规则                      | 角色动作和战斗逻辑              |
| `AWuwaPlayerController`    | 键鼠 Input Mapping、设备输入转语义命令、把命令交给 Pawn | 判断动作合法性、播放 Montage    |
| `AWuwaPlayerCameraManager` | 持有 Camera Mode Stack、计算最终 POV                    | 决定角色能否进入某动作          |
| `AWuwaCharacter`           | 组件装配、对外角色门面、Definition 初始化               | 堆积每项功能的实现细节          |
| `AWuwaEnemyCharacter`      | 敌人组件装配、敌人 Definition 初始化                    | 把敌人攻击规则硬编码在 Actor 中 |

`AWuwaCharacter` 是 **Composition Root**。它创建并连接组件，但行为实现留在组件内部。角色仅保留少量稳定 API，例如 `InitializeFromDefinition`、`GetActionRouter` 和 `GetCharacterStateView`。

### 3.3 接口

```text
IWuwaActionSource
    BuildActionRequest：把不可变 Input Command 转换为 Action Request，不暴露具体 Executor

IWuwaActionExecutor
    SupportsAction / CanStartAction / StartAction / EndAction

IWuwaTargetableInterface
    提供目标点、存活状态、阵营和可锁定状态

IWuwaDamageableInterface
    接受结构化 Damage Spec，不依赖具体 Character 类

IWuwaGrappleTargetInterface

    向动画、UI、镜头提供只读状态快照
```

接口只放跨域所需的最小信息。若两个组件同属一个域且不会替换，直接使用清晰 API 比滥用接口更简单。

## 4. 输入、命令和动作生命周期

### 4.1 输入分两类

持续状态不进入普通 FIFO 缓存：

```text
Move Vector（WASD）
Look Vector（鼠标）
Target Switch Axis（键盘输入非零时转换为离散命令）
```

离散命令进入动作请求/缓存：

```text
Jump Pressed
Sprint Pressed（同一按键的边沿事件，空中映射为二段跳请求）
Dodge Pressed
Attack Pressed
Grapple Pressed
Lock Toggle Pressed
Switch Character Pressed
```

输入适配规则：

- 项目只处理键盘与鼠标，不为摇杆死区、半推输入、手柄按键图标或设备热切换增加分支。
- `FVector2D` 仍作为 Move/Direction 的语义结构，便于动作分流、相机空间换算和对角方向归一化；它不代表必须支持模拟摇杆，也不要求所有动作都把 Direction 作为实际位移方向。
- 离散命令的 `Direction` 在按键边沿发生时从当前 WASD Move Intent 快照，命令入队后不得被后续输入改写。
- Sprint 不保存 Held/Released Gameplay 状态，只由 `ETriggerEvent::Started` 生成一次 Pressed Command；按键释放不取消动作或 Sprint Run。
- Direction 为零表示没有按下 WASD，不增加手柄死区参数。
- `Block.Input.Move` 只阻止 Character 将持续 Move Intent 写入 Movement/AddMovementInput，不清空 Controller 保存的真实 WASD 快照；来源句柄释放后的下一帧按当前键位恢复移动。Look 输入不受该标签影响。

建议从现有 `FWuwaInputIntent` 增量演进：保留持续快照，同时将单帧布尔值转换为结构化命令。

```cpp
USTRUCT(BlueprintType)
struct FWuwaInputCommand
{
    GENERATED_BODY()

    FGameplayTag InputTag;
    double PressedAt = 0.0;
    float ValidDuration = 0.15f;
    FVector2D Direction = FVector2D::ZeroVector;
    uint32 Sequence = 0;
};
```

### 4.2 动作请求与结果

```cpp
USTRUCT()
struct FWuwaActionRequest
{
    GENERATED_BODY()

    FGameplayTag ActionTag;
    TObjectPtr<UWuwaActionDefinition> Definition;
    FWuwaActionContext Context;
    uint32 SourceInputSequence = 0;
};
```

请求结果必须能解释：

```text
Started
Buffered
Rejected.InvalidDefinition
Rejected.MissingRequiredTag
Rejected.BlockedByTag
Rejected.Priority
Rejected.CancellationRule
Rejected.Cooldown
Rejected.InvalidContext
Rejected.NoExecutor
Rejected.ExecutorFailed
```

### 4.3 统一生命周期

```text
Input Command
  → Action Source Build Request/Context
  → Request / Validate Definition
  → CanStart
  → Resolve Executor / Executor CanStart
  → Resolve Conflict / Cancel Current
  → Acquire Granted Tags / Executor Start
  → Open/Close Windows and Update
  → Finish(Completed | Cancelled | Interrupted | Failed | OwnerDestroyed)
  → Executor EndAction / Release Granted Tags / Clear Runtime
  → Try Consume Buffered Command
```

Router 当前运行态保存：

- Definition / Executor Object
- ActionTag / Action Context / Source Input Sequence
- Granted Tag Handles

具体 Executor 按动作需要记录自己取得的资源句柄：

- Input Block Handles
- Camera Mode Request Handle
- Root Motion Source ID / Motion Warp Target 名称
- 临时重力、旋转或碰撞覆盖 Handle
- Montage Instance/Delegate Handle

退出时按句柄释放，避免“一个动作移除另一个动作添加的标签”或重置错误参数。

### 4.4 仲裁规则

- 同一时间只有一个 **Exclusive Action**，例如 Attack、Dodge、Mantle、HitReact。
- Locomotion 不是普通 Exclusive Action；Walking/Falling/CustomMode 由 Movement Component 持有。
- Sprint 可作为非独占状态，受 Action Router 与移动条件共同约束。
- Camera、VFX、UI 不是动作，只接收动作和状态事件。
- 高优先级不代表无条件打断；Definition 还要满足 `CanCancelActions` / `CanBeCancelledBy` 或 Cancel Channel。
- 死亡、击飞等强制动作走 `ForceRequest`，但仍必须触发被中断动作的清理。

## 5. 状态所有权与单一权威

| 状态/数据                        | 唯一拥有者                       | 其他系统如何使用            |
| -------------------------------- | -------------------------------- | --------------------------- |
| Capsule 位置、速度、MovementMode | Movement Component               | 只读查询/事件               |
| 当前独占动作与结束原因           | Action Router                    | 只读快照/事件               |
| 输入缓存                         | Input Buffer                     | 查询、消费，不直接改数组    |
| Health/Posture/Stamina           | Attribute Component              | Modifier API/变化事件       |
| 当前软锁/硬锁目标                | Targeting Component              | Target Context/事件         |
| 最终相机 POV                     | Player Camera Manager            | 提交 Mode Request           |
| Mesh、Anim Class、动作集         | Character Definition + Presenter | 初始化/切换时应用           |
| Montage 播放                     | 对应动作 Executor                | AnimInstance 只执行表现命令 |
| Gameplay Tags 聚合结果           | State Tag Component              | 只读 Tag Query              |

### 5.1 位移策略

每个动作的 Definition 明确 `MovementPolicy`：

```text
CharacterMovement       常规地面、空中、跑墙、攀爬、钩锁
RootMotionMontage       固定距离闪避或小型攻击（只能选定后使用）
MotionWarpedRootMotion  翻越、有限攻击吸附
RootMotionSource        需要代码可控、可组合的短时推进
None                    原地动作
```

同一动作只允许一个主位移权威。Motion Warping 只修正动画 Root Motion 目标，不与 `SetActorLocation` 或 `LaunchCharacter` 同帧争抢 Capsule。

### 5.2 临时覆盖采用“保存与恢复”或句柄栈

重力、旋转策略、移动输入缩放、碰撞响应和相机模式都不能用散落的“设为默认值”清理。临时动作开始时记录原值或取得覆盖句柄，结束时释放，最终值由剩余覆盖重新计算。

### 5.3 二段跳分流与落地事实

Sprint Pressed 是带方向快照的单个语义命令，由 MovementMode 和 WASD Direction 构建上下文动作：

```text
Sprint Command
├─ MovingOnGround + Direction != Zero → Ground Dash Forward
├─ MovingOnGround + Direction == Zero → Ground Backstep
├─ Falling + Direction != Zero        → Directional Air Sprint Double Jump
└─ Falling + Direction == Zero        → Backflip Double Jump
```

空中两条路径的规则：

- 空中 `Direction` 只选择二段跳类型：任意 W/A/S/D 或斜向输入都进入 Directional，完全零输入才进入 Backflip。
- 定向冲刺二段跳快照触发瞬间的 `ActorForward` 作为唯一水平推进方向；具体按键方向与相机朝向不再改变该次二段跳轨迹。
- 后空翻快照按下瞬间的角色朝向，以 `-ActorForward` 产生真实向后位移，但角色 Yaw 从开始到结束保持不变。
- 两者共用 `MaxJumpCount`、`bAirSprintConsumed` 和同一次空中预算；任一路径成功后，本次滞空不能再走另一条。
- Movement Component 是速度和跳跃次数权威；Action Router 只批准请求和管理动作生命周期。
- 两个二段跳 Definition 都授予 `Block.Input.Move`。Character 在标签存在期间提交零移动意图并跳过 `AddMovementInput`；Controller 继续保存真实 Move Intent，因而正常 Montage End 后能立即恢复空中控制。
- 二段跳 Montage 完整结束只作为正常完成的时间事件：Executor 通知 Router `FinishCurrent(Completed)`，Router 再按来源句柄释放 `Block.Input.Move`。Montage、AnimBP 或 Notify 不得直接移除标签。
- Cancelled、Interrupted、Failed、提前 Landed 和 OwnerDestroyed 不等待 Montage 原定结束时间，必须立即通过同一 Action 清理路径释放输入阻止与 Executor 资源。
- 后空翻保持朝向所需的旋转覆盖必须有明确来源，Completed、Cancelled、Interrupted、Failed、Landed 和 OwnerDestroyed 均释放。

落地事实只描述物理冲击：

```text
PreImpact Velocity.Z
→ ImpactSpeed = Max(0, -Velocity.Z)
→ 与 Movement Profile 的 HeavyLandingVelocityThreshold 比较
→ LightLanding 或 HeavyLanding
```

- `Light/Heavy` 只表示冲击速度等级，不表示攻击类型。
- `LandingType` 与 `LandingSource` 正交；Source 可以在未来说明 Normal/Plunge 等来源，但不能强制伪造 Heavy。
- 下落攻击是独立的 Plunge Action、Context、Executor 和 Attack Instance 结算链，不以 HeavyLanding 作为入口或攻击条件。
- Plunge 可以消费同一个真实 `Landed` 事实做一次结算；如果 Plunge 拥有专用落地 Montage，只抑制普通落地表现，不改写 ImpactSpeed、LandingType 或 Landing Sequence。

### 5.4 Day 5 冲刺位移权威

- 地面前冲/后撤统一使用 `RootMotionSource`，Root Motion Source ID 由当前动作实例持有并在所有退出路径移除。
- 地面前冲与空中定向二段跳统一快照触发瞬间的 `ActorForward`；地面后撤与后空翻统一使用 `-ActorForward`。Input Direction 只负责选择正向/后向分支，不作为实际位移方向。
- 地面冲刺动画使用 In-Place Montage；禁止同时使用 Montage Root Motion、`SetActorLocation` 或 `LaunchCharacter`。
- 定向二段跳与后空翻使用 `CharacterMovement` 的配置化速度，不添加第二个位移源。
- 前冲在 Montage 资产标记的中段 Handoff Point 重新读取 Character 保存的真实 WASD；有输入时由 Executor 校验后提前完成 Dash、保留出口速度，并在 Router 释放 Granted Tags 后进入短暂 Sprint Run；无输入时继续后半段恢复动画。Notify 只提供时间事件，不直接写 Gameplay 状态。
- Sprint Run 是出口减速状态，以 `RunSpeed` 为收敛目标，并由 Movement Profile 的 `SprintRunDeceleration` 独立控制水平减速；该阶段 `MaxWalkSpeed` 跟随当前衰减速度，防止 CharacterMovement 内置超速制动抢先钳回 RunSpeed。持续按键不能无限保持，后撤和外部取消、离地中断、失败等路径均不能进入。
- 地面动作从 Walking 转入 Falling 时使用一次性 `MaintainLastRootMotionVelocity` 保留水平惯性，避免边缘离地后突然零速。当前 Root Motion Source 依据整帧 `DeltaTime` 准备 Override Velocity，因此低帧率下继承的帧平均速度会使边缘飞出距离略大。该差异已接受为延后的非阻塞问题；未来若修正，应在地面动作离地出口建立帧率无关的水平速度策略，不通过 Montage 或曲线反向补偿。

## 6. 数据驱动设计

### 6.1 数据资产分层

```text
UWuwaCharacterDefinition : UPrimaryDataAsset
├── Identity / Mesh / AnimClass
├── MovementProfile
├── TraversalProfile
├── CombatProfile / AttackSet
├── TargetingProfile
├── CameraProfile
└── Ability/Action Definition References
```

具体数据资产：

| 数据资产                   | 配置内容                                                              | 主要消费者               |
| -------------------------- | --------------------------------------------------------------------- | ------------------------ |
| `UWuwaInputConfig`         | IMC 与 Input Action 引用                                              | PlayerController         |
| `UWuwaMovementProfile`     | 走/跑/冲刺速度、加减速、跳跃、定向/后空翻速度、空中控制、落地速度阈值 | Movement Component       |
| `UWuwaTraversalProfile`    | Trace 尺寸、墙角度、跑墙/攀爬/钩锁参数、耐力消耗                      | Traversal + Movement     |
| `UWuwaTargetingProfile`    | 搜索范围、评分权重、遮挡、保持与解除阈值                              | Targeting Component      |
| `UWuwaCameraProfile`       | 各模式 FOV、距离、偏移、Lag、Blend                                    | Camera Manager           |
| `UWuwaActionDefinition`    | 标签条件、优先级、缓存、取消、位移策略、Montage                       | Action Router + Executor |
| `UWuwaAttackDefinition`    | 伤害、韧性、Trace、吸附、Hit Stop、可弹刀属性                         | Combat Component         |
| `UWuwaAttackSet`           | 地面/空中/反击/入场攻击的动作映射                                     | Combat Component         |
| `UWuwaEnemyDefinition`     | 属性、攻击集、AI 参数、表现资源                                       | Enemy Character/AI       |
| `UWuwaCharacterDefinition` | 角色的组合配置入口                                                    | Character/Party          |

### 6.2 Data Asset、Data Table 与 Curve 的使用规则

- **Primary Data Asset**：有身份、由多个对象组合引用、需要独立验证的配置，如 Character/Enemy Definition。
- **Data Asset**：一组内聚参数或资源，如 Movement/Camera/Action Definition。
- **Data Table**：大量同构且适合批量比较的数据，如攻击派生数值或反馈强度；不要用一张巨表承载所有系统。
- **CurveFloat/CurveVector**：随归一化时间、速度或距离连续变化的数据，如钩锁速度、FOV、制动倍率。
- **Gameplay Tags**：语义标识、条件查询和映射键，不承载数值。

### 6.3 配置覆盖顺序

```text
C++ 安全默认值
  < 全局项目默认 Profile
  < Character/Enemy Definition 引用的 Profile
  < 当前动作 Definition 的短时覆盖
  < Debug 开发覆盖（仅非 Shipping）
```

运行时逻辑不直接修改 Data Asset。初始化时读取只读配置，临时变化保存到 Runtime State。

### 6.4 数据校验

每个关键 Data Asset 实现编辑器校验或统一 `ValidateConfig`：

- 必填引用不为空。
- Tag 类型正确，例如 `Action.*` 不误填为 `State.*`。
- 数值范围合法，Blend 时间、缓存时间和距离不为负。
- Heavy Landing 阈值为正，后空翻向上/向后速度与朝向锁定时间处于合法范围。
- MovementPolicy 与 Montage Root Motion 配置一致。
- Combo 下一段存在且不能形成非法循环。
- Character Definition 引用的 Skeleton/AnimClass 兼容。

校验失败在加载时输出资产路径和字段名；打包前执行一次全量资产验证。

## 7. 各功能域设计

### 7.1 Locomotion

`UWuwaCharacterMovementComponent` 负责：

- 读取 Movement Profile 并应用基础参数。
- Walking/Falling/Custom Movement 的物理更新。
- 普通跳跃、定向冲刺二段跳、后空翻二段跳的速度、次数预算和落地重置。
- 在父类落地处理改写速度前快照 Impact Velocity，并按 Profile 阈值生成 Light/Heavy Landing 事实。
- 速度、加速度、接地、落地速度等只读快照。
- 跑墙、攀爬、抓边、钩锁的 `PhysCustom` 分发。

它不主动做多方向环境搜索。Traversal Component 生成经过验证的 `FWuwaTraversalContext`，Movement Component 只消费当前动作所需的墙面/边缘/钩锁上下文并执行物理。

### 7.2 Traversal

Traversal 分成三步：

```text
Query：世界检测，得到原始命中
Evaluate：按 Profile 与角色状态生成候选和失败原因
Execute：向 Action Router 请求动作，成功后把稳定 Context 交给 Movement
```

`FWuwaTraversalContext` 在动作开始时快照关键数据，避免每个系统重复 Trace；运行中允许 Movement 按需刷新墙面接触，但原始进入条件不被悄悄改写。

### 7.3 Targeting

- 候选对象只依赖 `IWuwaTargetableInterface`。
- 收集、过滤、评分、保持/解除使用不同函数，便于显示失败原因。
- 软锁供攻击吸附参考；硬锁控制角色朝向与镜头构图；两者共享候选服务但状态独立。
- 切换目标使用屏幕空间左右关系和滞回阈值，防止来回跳变。
- 目标死亡/销毁通过弱引用和事件解除，不让 Combat 或 Camera 各自保留不安全裸指针。

### 7.4 Combat

```text
Input Command
→ Attack Action Request
→ Attack Definition / 当前 Combo Index
→ Action Router 准入
→ Combat Executor 播放 Montage
→ Window Event 打开 Trace / Combo / Cancel
→ Trace 产生 Hit Candidate
→ Combat Resolver 处理闪避、伤害、弹刀、韧性
→ Attribute Component 应用结果
→ Combat Event 驱动动画、相机、VFX、音频和 UI
```

伤害使用结构化数据，不直接调用“扣血数字”：

```text
Instigator / Causer
Attack Tag
Damage / Posture Damage
Hit Location / Normal
Hit Direction
Reaction Type
CanParry / CanPerfectDodge
Unique Attack Instance ID
```

`Unique Attack Instance ID + Target` 用于单段命中去重。

### 7.5 Animation

`UWuwaAnimInstance` 从 `IWuwaCharacterStateView` 拉取只读状态：速度、方向、接地、MovementMode、当前动作、目标方向等。

二段跳动画只读取 `LastJumpType/JumpSequence` 区分 Directional Air Sprint 与 Backflip；落地动画只读取 `LandingType/LandingSequence` 区分 Light/Heavy。二段跳 Montage End 可以向 Executor 报告时间结束，但只有 Router 的 Action Finish 才能释放 `Block.Input.Move`。动画不得根据 Heavy Landing 触发下落攻击，也不得修改 JumpCount、Velocity、角色朝向、LandingType 或 Gameplay Tag。

Anim Notify/Notify State 只发送时间事件：

```text
Window.AttackActive Open/Close
Window.Combo Open/Close
Window.Cancel.Dodge Open/Close
Window.Invincible Open/Close
Window.PerfectDodge Open/Close
Window.Parry Open/Close
MotionWarp Target Refresh（仅请求）
```

Notify 不直接扣血、不直接切 MovementMode、不直接永久添加状态。Montage 被异常中断时，动作 `Finish` 必须关闭仍打开的窗口。

### 7.6 Camera

Camera Mode 是数据，不是 Character 分支：

```text
ModeTag
Priority
FOV / ArmLength / Offsets
BlendIn / BlendOut
Lag Settings
Composition Policy
```

各系统通过来源句柄提交 Mode Request；Camera Manager 选择最高优先级并混合。来源结束或对象销毁时请求自动移除。镜头读取 Targeting 的目标点，但不能改变锁定状态。

### 7.7 Party 与多角色

不为第二角色复制组件和 C++ 类。`UWuwaPartyComponent` 持有 Character Definition 列表，切换是一个可取消事务：

```text
Validate → Lock Input → Preserve Shared Context
→ Exit Old Presentation → Apply New Definition
→ Restore Target/Camera → Entrance Action → Unlock
```

四周范围内优先复用同一个 Pawn 并切换 Definition/Mesh/Anim/数据，避免 Spawn/Destroy 两套 Pawn 带来的相机、输入和目标迁移复杂度。只有资源或动画结构不允许时才升级为多 Pawn 切换。

## 8. 事件通信规则

建议使用原生 Multicast Delegate 承载本地运行时事件：

```text
OnActionStarted / OnActionEnded / OnActionRejected
OnMovementModeChanged
OnTraversalCandidateChanged
OnTargetChanged
OnCombatEvent
OnAttributeChanged / OnDeath
OnCharacterDefinitionChanged
```

规则：

- 事件描述已经发生的事实；需要批准的操作必须调用请求 API。
- 委托参数使用结构体，避免以后不断修改签名。
- 订阅者不能在回调中偷偷改写事件源内部数组。
- 高频每帧数据由只读 View 拉取，避免广播速度/方向这类高频事件。
- 跨关卡全局 Event Bus 暂不引入；当前纵向切片不需要。

## 10. 调试、测试与质量门槛

### 10.1 Debug Read Model

Debug HUD 不逐个强依赖所有组件，而是从角色聚合一个只读快照：

```text
MovementMode / CustomMode / Velocity
CurrentAction / ActionTime / EndReason
ActiveTags + Source Count
BufferedCommands + RemainingTime
Traversal Context + Failure Reason
SoftTarget / HardTarget + Score Breakdown
Camera Mode Stack
Combat Windows / Attack Instance ID
Health / Posture / Stamina
```

### 10.2 自动化与手工测试

- 纯规则函数（目标评分、动作准入、缓存过期、伤害解析）尽量写 Automation Test。
- 物理与动画衔接使用小型 Dev Map 和固定重现场景。
- 每项动作至少测试：正常完成、主动取消、受击中断、掉出环境、Owner 销毁。
- 每日冒烟链：使用键盘与鼠标进入关卡、移动、跳跃、当天功能、停止 PIE、无 Error。
- 每周末在 Development 打包运行，不把“编辑器可用”当成完成。

### 10.3 Definition of Done

单项功能只有同时满足以下条件才算完成：

- 核心路径可运行，失败路径有明确结果。
- 关键参数已从 C++ 魔法数字迁移到 Profile/Definition。
- 动作异常结束能恢复 MovementMode、重力、碰撞、旋转、输入和相机。
- Debug HUD 或日志能解释当前状态与拒绝原因。
- 至少有一个可重复测试场景。
- 没有新增对 `Variant_*` 示例业务代码的依赖。
- 编译通过并完成一次小范围回归。

## 11. 推荐实现顺序

```text
现有 Day 1/2 基线
→ Gameplay Tags + Profile 基础设施
→ Input Command / Buffer / Action Router
→ Locomotion + Targeting + Camera
→ Traversal Query/Context + Custom Movement
→ Combat Definition/Resolver
→ Enemy + Dodge/Parry
→ Character Definition + Party
→ 表现、稳定性、打包
```

这个顺序让“数据驱动”和“统一动作生命周期”在第一周落地，后续探索、战斗和角色切换都复用同一边界，而不是到第四周再迁移硬编码。

## 12. 架构守则检查表

新增功能前检查：

- [ ] 谁拥有这份状态？是否只有一个权威？
- [ ] 这是请求、事实事件，还是只读查询？
- [ ] 是否能用现有组件的窄 API，而不是新增双向依赖？
- [ ] 数值与资源是否应进入 Profile/Definition？
- [ ] 动作使用哪一种位移策略？
- [ ] 正常结束、取消、中断、销毁分别如何清理？
- [ ] 动画和镜头是否只是状态消费者？
- [ ] Debug 如何解释进入失败或状态卡住？
- [ ] 第二角色是否只需替换数据，而不复制代码？
